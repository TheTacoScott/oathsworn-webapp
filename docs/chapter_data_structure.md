# chapters.js Data Structure

`web/data/chapters/` is generated by `scripts/generate_data.py` from the
decompiled Android source. Each chapter has its own file:

```
web/data/chapters/init.js        declares const CHAPTERS = {}
web/data/chapters/chapter_1.js   CHAPTERS[1]  = { ... }
web/data/chapters/chapter_22.js  CHAPTERS[22] = { ... }  (chapter 11.5)
...
```

The internal chapter number is used as both the filename suffix and the key
into `CHAPTERS`. Chapter 11.5 is always `22` internally.

---

## Chapter object

```js
CHAPTERS[n] = {
  "num":                    <int>,
  "clue":                   <int[]>,
  "clueLocation":           <int>,
  "clueLocationSectionNum": <int>,
  "location":               { "<locationId>": <sectionIndex>, ... },
  "deepwoodChapter":        <bool>,
  "deepwoodTokens":         <int[]>,
  "deepwoodMap":            { "<sectionIndex>": <sectionIndex>, ... },
  "timeTriggers":           { "<timeValue>": <sectionIndex>, ... },
  "onNormalPath":           <"always" | "never" | { "type": "ranges", "ranges": [[start, end], ...] }>,
  "sections":               [ <Section | null>, ... ]
}
```

### `num`
The internal chapter number. Matches the key used to store this chapter in
`CHAPTERS`. Chapter 11.5 is stored as 22 everywhere internally.

### `clue`
Array of up to two section indices that are "clue" locations. When the player
visits both sections listed here, the `clueLocation` is automatically added to
the active locations list, unlocking a hidden path.

- Default `[-1]` means no clue mechanic in this chapter.
- `clue[0]` is the first clue section, `clue[1]` is the second.
- Both must be visited (in any order) to trigger the unlock.

### `clueLocation`
The location ID that gets added to the locations list once both clue sections
are visited. Displayed as a location button the player can then navigate to.
`-1` means unused.

### `clueLocationSectionNum`
The section index that the `clueLocation` button navigates to. Also used
during back-navigation: if the player backs out of this section, the clue
location is re-added to the removed list. `-1` means unused.

### `location`
Maps location ID (as a string-coerced integer) to the section index that
location button navigates to. All location buttons visible in a given section
look up their destination here.

```js
"location": { "101": 5, "102": 12 }
// Location 101 navigates to section 5, location 102 to section 12
```

Location IDs are chapter-namespaced integers. Their display label is derived
by stripping the leading chapter-prefix digit(s): e.g. `101` displays as `"1"`,
`1101` displays as `"01"`.

### `deepwoodChapter`
`true` for chapters 4, 10, 14, 17, and 18, which use the token-exploration
system. `false` for all others.

### `deepwoodTokens`
Array of section indices that are deepwood token locations. On a player's
first visit to one of these sections, it is marked visited. On a second visit,
`deepwoodMap` redirects to an alternate section instead.

### `deepwoodMap`
Maps a deepwood token section index to its "second visit" alternate section
index. Only relevant when `deepwoodChapter` is `true`.

```js
"deepwoodMap": { "15": 42, "23": 67 }
// Second visit to section 15 goes to section 42 instead
```

### `timeTriggers`
Maps a cumulative time-track value to a redirect section index. When the
player's time counter reaches exactly one of these values (as a result of
`timeAdded` on a section), the engine sends the player to the redirect section
rather than the chosen destination.

```js
"timeTriggers": { "5": 30, "12": 61 }
// Reaching time 5 redirects to section 30; reaching time 12 to section 61
```

The player's `nextPositionToken` is saved before the redirect so they can
return to the main story afterward (via a choice with `next: -3`).

### `onNormalPath`
Controls which sections are considered "on the normal (main) story path" for
the purpose of saving `nextPositionToken`. Before a time-track trigger fires,
the engine saves the intended destination as the return point - but only if
that destination is on the normal path.

- `"always"` - every section is on the normal path (most chapters)
- `"never"` - no section is on the normal path
- `{ "type": "ranges", "ranges": [[start, end], ...] }` - sections whose
  index falls within any of the given inclusive ranges are on the normal path

### `sections`
Zero-indexed array of Section objects (see below). The array index IS the
section number used in `choices[i].next`, `location` values, etc.

A `null` entry means the section at that index failed to parse from the source
(should not happen with a clean APK decompile).

---

## Section object

Each non-null entry in `sections` looks like this:

```js
{
  "popUpTexts":             [<stringKey | null>, <stringKey | null>, <stringKey | null>, <stringKey | null>],
  "audio":                  [<audioKey | null>,  <audioKey | null>,  <audioKey | null>,  <audioKey | null>],
  "sectionTexts":           [<stringKey | null>, <stringKey | null>, <stringKey | null>, <stringKey | null>],
  "locationsAdded":         <int[]>,
  "showLocations":          <bool>,
  "isLocation":             <bool>,
  "clearLocationsList":     <bool>,
  "removeSpecificLocations":<int[]>,
  "timeAdded":              <int>,
  "imageLinks":             <string[] | null>,
  "imagePositions":         <int[] | null>,
  "choices":                <Choice[] | null>
}
```

---

### `sectionTexts`
Array of exactly 4 slots. Each slot is either a key into `STRINGS` (the
string to display) or `null` (slot unused).

Slots render as plain story-text paragraphs. They are interleaved with
`popUpTexts` and images according to `imagePositions` (see Plate Rendering
below).

```js
"sectionTexts": ["chp1_21_1__a", "chp1_21_1__c", "choose_one", null]
// Slot 0: first paragraph of story text
// Slot 1: second paragraph
// Slot 2: a generic "choose one" prompt
// Slot 3: unused
```

---

### `popUpTexts`
Array of exactly 4 slots. Each slot is either a key into `STRINGS` or `null`.

Slots render as visually distinct "popup boxes" - styled differently from
normal story text to draw attention (rules reminders, special instructions,
etc.). They are interleaved with `sectionTexts` according to the plate
rendering order.

```js
"popUpTexts": ["pop1_21_1__b", null, null, null]
// Slot 0: one popup box is shown
// Slots 1-3: unused
```

---

### `audio`
Array of exactly 4 slots. Each slot is either an audio file base name
(resolved to `data/audio/<name>.mp3`) or `null`.

Non-null slots play in sequence. When the current track ends, playback
advances to the next non-null slot automatically.

**Special case:** if `audio[0]` is the literal string `"encounter_audio"`,
the track loops continuously (used for encounter background music) rather
than advancing.

```js
"audio": [null, "chp1_21_1__c", null, null]
// Slot 0: unused
// Slot 1: plays data/audio/chp1_21_1__c.mp3
// Slots 2-3: unused
```

---

### `imageLinks`
Array of drawable resource base names, or `null` if no images. Each name
resolves to `data/images/<name>.<ext>` where the extension comes from
`IMAGE_EXT[name]` (either `"jpg"` or `"png"`).

The number of entries must match the length of `imagePositions`. Both arrays
are parallel: `imageLinks[i]` is placed at the position `imagePositions[i]`.

```js
"imageLinks": ["ch1_21_1__p4"]
// One image: data/images/ch1_21_1__p4.jpg (or .png)
```

---

### `imagePositions`
Array of integers (one per entry in `imageLinks`) that control where each
image appears within the plate's rendered content. `null` if no images.

Each position value maps to a slot in the interleaved plate layout (see
Plate Rendering below). Valid values are 1-8.

```js
"imagePositions": [4]
// The image in imageLinks[0] appears at position 4 (after sectionTexts[1])
```

---

### Plate Rendering Order

The visible content of a section is rendered by sorting all items by a
computed key and displaying them top-to-bottom. This is how `sectionTexts`,
`popUpTexts`, and images are interleaved:

| Sort key | Content source         | Notes                          |
|----------|------------------------|--------------------------------|
| 1        | image at position 1    | very top of plate              |
| 2        | `sectionTexts[0]`      |                                |
| 3        | image at position 2    |                                |
| 4        | `popUpTexts[0]`        |                                |
| 5        | image at position 3    |                                |
| 6        | `sectionTexts[1]`      |                                |
| 7        | image at position 4    |                                |
| 8        | `popUpTexts[1]`        |                                |
| 9        | image at position 5    |                                |
| 10       | `sectionTexts[2]`      |                                |
| 11       | image at position 6    |                                |
| 12       | `popUpTexts[2]`        |                                |
| 13       | image at position 7    |                                |
| 14       | `sectionTexts[3]`      |                                |
| 15       | image at position 8    | after last text, before popup  |
| 16       | `popUpTexts[3]`        | very bottom of plate           |

Formula: image sort key = `imagePositions[i] * 2 - 1`

Items with a `null` value are omitted entirely. Only non-null items appear.

**Example** using the section at the top of this doc:
- sectionTexts[0] at key 2
- popUpTexts[0] at key 4
- sectionTexts[1] at key 6
- image (position 4) at key 7
- sectionTexts[2] at key 10

Rendered top-to-bottom: text, popup box, text, image, text.

---

### `locationsAdded`
Array of location IDs to add to the player's active locations list when
entering this section. `[-1]` means nothing is added.

Added locations are sorted numerically and displayed as buttons when
`showLocations` is `true` on any section.

```js
"locationsAdded": [101, 102]  // Adds locations 101 and 102
"locationsAdded": [-1]        // Nothing added
```

### `showLocations`
`true` if this section should display the current active locations list as
clickable buttons below the story text. The locations shown are whatever IDs
are currently in the player's `locationsList` state.

Typically `true` on "hub" sections where the player chooses where to explore.

### `isLocation`
`true` if this section itself is a location tile (i.e., the player arrived
here by clicking a location button). Used exclusively by the back-navigation
logic: when undoing a move away from this section, the location is re-added
to the active list.

Does not affect forward navigation or rendering.

### `clearLocationsList`
`true` if all currently active locations should be cleared when entering this
section. The cleared locations are pushed onto a removal stack so they can be
restored if the player navigates back.

### `removeSpecificLocations`
Array of specific location IDs to remove from the active locations list when
entering this section. `[-1]` means nothing is removed.

```js
"removeSpecificLocations": [101]  // Remove location 101
"removeSpecificLocations": [-1]   // Nothing removed
```

### `timeAdded`
Integer amount added to the chapter's time track when the player leaves this
section by clicking a choice or location button. `-1` means no time is added
(the time track does not advance).

After time is added, the engine checks `timeTriggers` to see if the new total
triggers a redirect.

```js
"timeAdded": 2   // Adds 2 to the time counter when leaving
"timeAdded": -1  // No time added
```

### `choices`
Array of Choice objects representing the navigation buttons shown at the
bottom of the screen, or `null` if there are no choices (rare; usually means
the chapter ends here or only location buttons are shown).

Each choice:

```js
{
  "text": <stringKey>,  // Button label, looked up in STRINGS
  "next": <int>         // Destination section index, or a sentinel value
}
```

**Sentinel values for `next`:**

| Value | Meaning |
|-------|---------|
| `-1`  | Chapter complete - return to chapter select screen |
| `-2`  | Player died - clear all chapter state and restart from section 0 |
| `-3`  | Return to `nextPositionToken` - resume the main story path after a time-track detour |
| `0+`  | Normal navigation to that section index |

```js
"choices": [
  { "text": "btn1_21_1__a", "next": 57 },  // Go to section 57
  { "text": "btn1_21_1__b", "next": 58 }   // Go to section 58
]
```

## String key naming conventions

String keys (used in `sectionTexts`, `popUpTexts`, `choices[i].text`) follow
a consistent pattern derived from the original Android resource names:

| Prefix   | Usage             | Pattern example        |
|----------|-------------------|------------------------|
| `chp`    | Story text        | `chp1_21_1__a`         |
| `pop`    | Popup box text    | `pop1_21_1__b`         |
| `btn`    | Choice button     | `btn1_21_1__a`         |
| `ch`     | Image drawable    | `ch1_21_1__p4`         |

The number segments are typically `chapter_section_plate__slot` but this
convention is not enforced anywhere in the code - treat string keys as opaque
identifiers looked up in `STRINGS`.

Some string keys are reused globals (e.g. `"choose_one"`, `"location_starter"`)
that appear across many chapters.

## Editing chapters.js safely

- **Finding a section:** Ctrl+F `[CH5]` to jump to chapter 5, then find the
  section by its array index (sections are one-per-line, so section 47 is
  the 48th line of the sections array, 0-indexed).

- **Changing a navigation target:** Edit `choices[i].next` to any valid
  section index within the same chapter, or a sentinel (-1, -2, -3).

- **Suppressing a time trigger:** In the chapter's `timeTriggers`, change the
  redirect value for that time step. Setting it to a section that just
  continues normally effectively disables the trigger.

- **Adding a location:** Add its ID to `locationsAdded` on the section where
  it should appear, and add a corresponding entry in the chapter's `location`
  map pointing to the destination section.

- **Re-generating:** Running `./setup.sh` will overwrite any manual edits to
  `web/data/chapters.js`. Keep a backup or re-apply edits after regenerating.
